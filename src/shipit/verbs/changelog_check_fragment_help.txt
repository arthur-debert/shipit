`shipit changelog check-fragment` fails a pull request to `main` that adds no
release-note fragment, so a missing fragment is caught at PR time — not at the
next release cut, once every PR is already merged.

It is self-gating and offline (no `gh` auth). Only the base ref comes from the
CI runner environment (`GITHUB_BASE_REF`, empty off a PR); everything else — the
added fragment and the escape hatch — is read from the PR's own git, so no CI
event trigger and no event payload is involved.

The check PASSES (exit 0), requiring no fragment, when any of:

- it is not a PR context (`GITHUB_BASE_REF` empty) — so laptop/lefthook runs
  never block
- the PR base is not `main` — work-stream PRs target epic branches and are
  exempt; only PRs to `main` are gated
- a commit on the PR carries a `Changelog: skip` trailer — the escape hatch for
  docs/CI/chore-only PRs that legitimately need no release note. A trailer (not
  a GitHub label): it rides the same git the gate already reads, so it works
  offline, on a re-run, and on a laptop, and never churns the CI suite the way a
  mutable label toggle would.

Otherwise it requires the PR's own diff (the three-dot merge-base diff against
`origin/main`, `--diff-filter=A`) to ADD at least one `CHANGELOG/unreleased-*.md`
fragment, and exits non-zero when none is found. A fragment the PR introduces
counts even when amended across review rounds (the base never had it); merely
modifying, deleting, or renaming a fragment already on `main` does not.

Wire it as a required, PR-only lane in `.shipit.toml` (`local = false`, so it
runs in CI but never on a laptop commit/push). The base-branch and label gating
lives here, in the subcommand — the lane planner stays base/label-unaware.
