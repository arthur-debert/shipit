# The PreToolUse guard's outer wrapper fails CLOSED, never open

> **Status: Accepted.** Fixes #529 (regression from #505/#491, PRs merged
> 2026-07-06). Scoped amendment to ADR-0012 (the guard) and ADR-0033 (hooks ride
> the pin via `./bin/shipit`) — narrower than either: it touches only the
> managed `PreToolUse` command's OUTER shell wrapper, not the guard's Python
> logic (`shipit.harness.policy`, unchanged) nor the other four managed hooks
> (`sessionstart`/`stop`/`subagent-stop`/`worktreecreate`, unchanged, still
> legitimately fail-open per #491).

## The regression

PR #505 (`289536f`, closing #491/#478 — "managed hook robustness in pixi-less /
other-machine envs") moved **every** managed `.claude/settings.json` hook,
`PreToolUse` included, onto one shared shape: drop the `pixi run` wrap (the
pinned `./bin/shipit` launcher is pixi-independent, ADR-0033) and fail open —
`cd "$CLAUDE_PROJECT_DIR" || exit 0; test -x ./bin/shipit || { echo ...; exit
0; }; ./bin/shipit hook <phase>`.

That shape is correct for the four ADDITIVE hooks it was designed for
(advisory/bookkeeping — a session-start staleness note, an eval log line, a
Tree-provisioning trigger): if the runtime is genuinely absent, skipping is the
right call, and it was already the ADR-0012 guard's OWN fail-open contract for
malformed *payloads* it received in-process. It is **wrong** for `PreToolUse`,
because that hook's job is not advisory: it is the sole enforcer of "the
coordinator never implements" (ADR-0012). A `PreToolUse` process never sources
`CLAUDE_ENV_FILE` (only Bash *tool calls* do), so it runs with a bare `PATH` and
no pixi activation — meaning the shared shape's guard-liveness silently
depended on ambient resolution. When resolution failed (as #504, `d0dd5ae`,
corroborates it did in this exact window — an unrelated launcher CRLF/pin-parse
fragility fix landed hours later), `exit 0` fired, Claude Code read "no
decision" as allow, and the coordinator ran **completely unguarded with zero
signal.** This is exactly how the LNT01 epic ran without ever cutting an epic
branch — the guard that should have redirected coordinator edits to a subagent
never fired.

## Decision

`PreToolUse`'s managed command is no longer generated by the shared
"additive-hook" shape. It gets its own, deliberately narrower contract:

1. **Restore `pixi run`, in the adapter-equivalent manifest-pinned form.** The
   command is
   `cd "$CLAUDE_PROJECT_DIR" && pixi run --manifest-path "$CLAUDE_PROJECT_DIR"/pixi.toml -- ./bin/shipit hook pretooluse`.
   `pixi run` supplies a resolvable, activated environment for a process that
   would otherwise see a bare `PATH`; the explicit `--manifest-path` (mirroring
   `shipit.pixienv.run.run_argv`) overrides any leaked `PIXI_PROJECT_MANIFEST` so
   the guard resolves THIS project rather than a parent's — the same ambiguity
   dodge the pixi adapter uses — and `--` fences pixi's flags from the child
   argv; `./bin/shipit` still supplies the pin (ADR-0033's "pin-wins, loudly" is
   unchanged — this is not a second pin story, just a working environment for the
   pin resolver to run in).
2. **No bare `exit 0` anywhere in this command.** The chain's own exit code is
   captured; on anything but `0` (the launcher missing/not executable, `cd`
   itself failing, the pin/uv/pixi chain unresolvable), the wrapper prints an
   actionable message to stderr and `exit 2`s — Claude Code's blocking-error
   exit code — so the tool call is refused. There is no code path in this
   command that both fails to run the real guard AND exits 0: the never-silent-
   allow invariant holds by construction, not by convention.
3. **"Ran and decided" is not "could not run".** When the chain DOES execute
   (rc 0), `shipit hook pretooluse` has already written ITS OWN decision
   straight to stdout — a `deny` JSON payload, or nothing for an allow — and
   its own exit code is always 0 by contract
   (`shipit.verbs.hook.pretooluse`). That inner fail-open-on-malformed-input
   contract is UNCHANGED and orthogonal: it answers "the guard ran; what does
   it make of this payload?", never "did the guard run at all?" — which is
   this ADR's exclusive concern. Conflating the two was the root mistake #505
   made; keeping them distinct is the fix.
4. **Scope: `PreToolUse` only.** The other four managed hooks keep the #491
   shape (`managed_cc_hook_command` in the test suite) untouched — they are
   additive, never the security guard, and #491's reasoning for them stands.

## Open point, deliberately left open

Restoring `pixi run` re-raises the pixi-less-consumer case #491/#478 targeted:
a repo with no pixi at all now gets `pixi run` itself failing (`rc=127`),
which — correctly, per the invariant above — BLOCKS every coordinator code
edit, rather than the pre-#529 silent allow. That is the right direction (fail
closed beats fail open, always), but it means a pixi-less consumer's guard
never actually RUNS; it only ever refuses. Making the guard genuinely live on
such a consumer (an alternate resolution path that does not require `pixi run`
but still cannot silently no-op) is left for a follow-up; this ADR does not
resolve it.

## Consequences

- The guard's liveness no longer depends on ambient `PATH` resolution outside
  an activated pixi env — the same coherence property ADR-0033 gives every
  other pinned invocation.
- A resolution failure is now loud (stderr message, non-zero exit) instead of
  silent — an operator or CI running the hook manually sees the refusal
  immediately, rather than a quiet allow with no trace.
- The template data file (`src/shipit/data/claude-settings-pretooluse.json`),
  shipit's own dogfood `.claude/settings.json`, and the `.shipit.toml`
  self-cert sha for the `PreToolUse` block move together in the same diff (the
  #505 precedent) — a drift among the three is caught by the reconcile-to-noop
  test coverage (`tests/test_install.py`).
